# 01 RecSys Overview：推荐系统总览

## 核心直觉

推荐系统的核心任务是：在海量物品中，为当前用户、当前场景选出最合适的一小批内容。

它通常不能直接用一个模型从全库物品里精排所有候选，因为物品规模太大、延迟要求太高。所以工业系统会分层：

```text
召回：从百万/亿级物品里快速捞几百到几千个候选
粗排：用轻量模型快速筛掉明显不合适的候选
精排：用复杂模型做高质量排序
重排：加入多样性、规则、业务约束
```

推荐系统真正难的地方不只在模型，而在样本、特征、延迟、反馈闭环和线上线下一致性。

## 关键概念

`用户`：推荐的接收者，可以有长期兴趣、短期兴趣、画像特征、实时行为。

`物品`：被推荐对象，例如视频、商品、文章、广告。

`上下文`：请求场景，例如时间、地点、设备、入口、网络状态。

`曝光`：系统把物品展示给用户。

`点击`：用户点击曝光物品。

`转化`：更深层目标，例如购买、收藏、关注、完播。

`负反馈`：不喜欢、拉黑、快速划走、投诉等显式或隐式信号。

`多目标`：CTR、CVR、GMV、留存、时长、满意度等目标经常互相拉扯。

## 必要公式 / shape / 流程

典型在线链路：

```text
request(user, context)
-> recall candidates
-> ranking scores
-> rerank with rules/diversity
-> show items
-> collect logs
-> build samples
-> train model
-> deploy model
```

CTR 预估：

```text
P(click = 1 | user, item, context)
```

CVR 预估：

```text
P(conversion = 1 | click, user, item, context)
```

多目标打分示意：

```text
score = w1 * CTR + w2 * CVR + w3 * value + w4 * diversity
```

样本构造：

```text
曝光且点击 -> 正样本
曝光未点击 -> 常见负样本
未曝光 -> 通常不能直接当负样本
```

## 代码阅读提示

读推荐系统代码或架构文档时，先定位它属于哪一层：

```text
召回：关注速度、覆盖、向量索引、候选规模
排序：关注特征、loss、AUC/LogLoss、模型表达
重排：关注业务规则、多样性、去重、频控
样本链路：关注曝光日志、label 定义、时间窗口
```

不要看到 DeepFM、DIN、双塔就直接背模型。先问它在链路中的位置：双塔多用于召回，DeepFM/DIN 多用于排序。

## 常见陷阱

- 把推荐系统理解成单个模型，而不是链路系统。
- 忽略曝光日志，只用点击日志构造样本，导致选择偏差。
- 用离线指标代表全部线上效果，忽略延迟、覆盖、多样性。
- 只优化 CTR，可能伤害转化、满意度或长期留存。
- 没区分召回和排序的目标，拿精排模型思路回答召回问题。

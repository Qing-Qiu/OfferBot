# 05 Model Serving：模型服务

## 核心直觉

模型服务解决的是：如何让模型在真实请求里稳定、低延迟、高吞吐地返回结果。

训练时你可以慢一点、批量大一点、资源多一点；在线服务时用户在等结果，系统必须考虑：

```text
延迟
吞吐
可用性
成本
降级
扩缩容
```

## 关键概念

`online inference`：实时推理，用户请求到来时马上算。

`batch inference`：批量推理，提前离线算好结果。

`REST`：基于 HTTP 的接口，简单通用。

`gRPC`：高性能 RPC，适合内部服务和强 schema 场景。

`缓存`：缓存特征、embedding、模型结果或候选集，降低重复计算。

`限流`：防止流量超过系统承载能力。

`熔断`：下游异常时快速失败，避免级联故障。

`降级`：模型不可用或超时时切换到规则、缓存、热门结果。

`热更新`：不中断服务加载新模型。

## 必要公式 / shape / 流程

在线推理链路：

```text
request
-> 参数校验
-> 特征获取
-> 特征预处理
-> 模型推理
-> 后处理
-> 返回结果
-> 记录日志
```

延迟分解：

```text
total latency =
network latency
+ feature latency
+ model inference latency
+ postprocess latency
+ queue waiting
```

吞吐：

```text
QPS = 每秒处理请求数
```

批处理权衡：

```text
batch size 大 -> 吞吐更高，但单请求等待可能更久
batch size 小 -> 延迟更低，但硬件利用率可能差
```

## 代码阅读提示

读模型服务代码时，先找：

```text
模型加载位置
请求 schema
特征获取函数
predict / infer 函数
批处理逻辑
异常处理
超时设置
日志和监控
降级逻辑
```

如果服务偶发慢，重点看：特征服务、队列等待、batch 聚合、GPU 利用率、下游依赖、冷启动加载。

## 常见陷阱

- 只优化模型计算，忽略特征获取才是主要延迟。
- 没有超时和降级，导致下游抖动拖垮主链路。
- batch 越大越好是错的，在线系统要权衡等待时间。
- 模型加载放在请求路径里，导致每次请求都重复加载。
- 日志记录太重，反过来拖慢服务。
- 没有 schema 校验，线上脏输入导致隐蔽错误。

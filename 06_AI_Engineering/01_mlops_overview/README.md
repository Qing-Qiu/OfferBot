# 01 MLOps Overview：MLOps 总览

## 核心直觉

MLOps 解决的是：模型不是训练完就结束，而是要被稳定、可观测、可回滚地放到线上，并持续迭代。

一个真实 AI 系统至少包含：

```text
数据采集
样本构造
训练
评估
部署
服务
监控
反馈回流
再训练
```

面试里，工程化问题往往不是问你会不会调一个 API，而是问你能不能把模型放进业务系统里，并在出问题时定位和兜底。

## 关键概念

`离线训练`：用历史数据训练模型，通常关注吞吐、成本、可复现。

`离线评估`：在验证集/测试集上评估模型，判断是否值得上线。

`在线服务`：实时响应请求，关注延迟、吞吐、可用性。

`batch inference`：批量离线推理，例如每天给全量用户生成推荐。

`online inference`：请求到来时实时推理。

`灰度发布`：先给小部分流量使用新模型，观察指标后逐步放量。

`回滚`：新模型异常时快速切回旧版本。

`监控`：同时监控服务指标、数据指标、模型指标和业务指标。

## 必要公式 / shape / 流程

标准链路：

```text
日志 -> 数据清洗 -> 样本构造 -> 特征生成
-> 训练 -> 离线评估 -> 模型注册
-> 灰度部署 -> 在线服务 -> 监控报警
-> 日志回流 -> 下一轮训练
```

核心指标：

```text
服务：latency, QPS, error rate, availability
模型：AUC, LogLoss, Recall, NDCG, calibration
数据：缺失率, 分布漂移, 异常值比例, 样本量
业务：CTR, CVR, GMV, 留存, 时长
```

上线判断：

```text
离线指标通过
回放测试通过
小流量灰度通过
核心业务指标不劣化
延迟和资源成本可接受
```

## 代码阅读提示

读工程化项目时先找配置和入口：

```text
训练入口: train.py / trainer
数据配置: dataset / feature config
模型定义: model / module
评估入口: evaluate.py
服务入口: app / server / predictor
部署配置: docker / yaml / ci
```

不要只看模型文件。MLOps 里，数据版本、特征版本、模型版本、配置版本同样关键。

1. 一个模型从训练到上线需要哪些步骤？
2. 离线训练和在线推理关注点有什么不同？
3. 新模型上线后指标变差怎么排查？
4. 如何做模型灰度和回滚？
5. 线上线下特征不一致怎么发现？
6. 数据漂移怎么监控？
7. 为什么需要模型版本管理？
8. 离线指标涨了，线上不涨可能为什么？

## 常见陷阱

- 只讲训练，不讲数据、部署、监控和回滚。
- 只监控服务延迟，不监控模型效果和数据分布。
- 训练数据和线上请求特征生成逻辑不一致。
- 没有模型版本和配置版本，问题难以复现。
- 灰度流量太小或时间太短，误判模型效果。
- 没有兜底策略，新模型异常会直接影响全量用户。

# 11 System Design Interview：AI 系统设计面试

## 核心直觉

AI 系统设计题不是让你背某个标准答案，而是看你能不能把业务目标、数据链路、模型链路、服务架构、指标实验和风险兜底串起来。

一个稳的回答框架：

```text
先问目标
再定指标
再画链路
再拆模块
再讲瓶颈
最后讲监控和兜底
```

## 关键概念

`业务目标`：系统真正优化什么，例如点击、转化、回答准确性、人工效率。

`数据链路`：日志从哪里来，样本怎么构造，特征怎么生成。

`模型链路`：用什么模型，怎么训练，怎么评估，怎么更新。

`服务架构`：线上请求如何经过各个服务，延迟和吞吐如何保证。

`指标体系`：离线指标、在线指标、业务指标、稳定性指标。

`灰度实验`：如何小流量验证新方案。

`风险兜底`：服务异常、模型异常、数据异常时如何降级和回滚。

## 必要公式 / shape / 流程

通用答题模板：

```text
1. Clarify: 业务目标、用户规模、延迟要求、数据来源
2. Metrics: 主指标 + 辅助指标 + 护栏指标
3. Data: 日志、样本、特征、数据质量
4. Model: baseline -> advanced model -> update cadence
5. Serving: API、缓存、特征服务、模型服务、降级
6. Experiment: A/B、灰度、显著性
7. Monitoring: 数据漂移、模型效果、服务稳定性
8. Risks: 冷启动、偏差、安全、成本
```

推荐系统模板：

```text
多路召回 -> 粗排 -> 精排 -> 重排 -> 曝光日志 -> 样本回流
```

RAG 系统模板：

```text
文档解析 -> chunk -> embedding -> 向量索引
query rewrite -> retrieve -> rerank -> prompt -> generate -> citation
```

向量召回模板：

```text
离线 item embedding -> ANN index
在线 user/query embedding -> topK search -> fusion/ranking
```

## 代码阅读提示

系统设计题虽然不一定写代码，但你要能把模块落到真实接口：

```text
数据任务: scheduled batch job / streaming job
模型训练: trainer + config + artifact
模型服务: predict(request) -> response
特征服务: get_features(entity_ids)
向量服务: search(vector, top_k, filters)
实验平台: assign_bucket(user_id)
监控系统: emit metrics/logs/traces
```

答题时避免只说“用深度学习模型”。要说清楚模型输入、输出、训练样本、上线方式和失败兜底。

## 面试高频问法

1. 设计一个短视频推荐系统。
2. 设计一个 RAG 企业知识库问答系统。
3. 设计一个双塔向量召回系统。
4. 设计一个广告 CTR 预估系统。
5. 设计一个在线特征服务。
6. 设计一个模型评测系统。
7. 新模型上线后效果下降怎么排查？
8. 如何处理冷启动、热门偏置和长尾问题？

## 常见陷阱

- 一上来就讲模型，没有先澄清业务目标和约束。
- 只画在线服务，不讲日志回流和训练闭环。
- 只讲离线指标，不讲 A/B 和护栏指标。
- 没有降级方案，系统不可用时无路可退。
- 忽略权限、安全、隐私和数据隔离。
- 只说“加缓存”，不说明缓存对象、失效策略和一致性风险。

## 自测题

1. AI 系统设计题开头应该澄清哪些信息？
2. 推荐系统的主指标和护栏指标可以怎么设置？
3. RAG 系统如何拆离线链路和在线链路？
4. 向量召回系统如何权衡召回率和延迟？
5. 模型上线后业务指标下降，你会按什么顺序排查？
6. 为什么系统设计必须讲监控和回滚？
7. 在线特征服务最怕什么一致性问题？
8. 如何把“模型”回答升级成“系统”回答？
